Write a SMALL, HIGH-SIGNAL set of integration tests that cover ONLY the main workflows and critical interactions between modules in this app.

Do NOT try to generate integration tests for every function or module. Focus on:
- Authentication and authorization flows (if present)
- Payment / billing / subscription flows (if present)
- Data creation / update / delete operations with meaningful side effects
- Any workflow where silent breakage would seriously harm users, money, or data

For each critical workflow:
- Create 1–3 integration tests that exercise the happy path, a key failure path, and (optionally) one edge case.
- Prefer fewer, more meaningful tests over broad but shallow coverage.

If an eval.json (or equivalent spec) exists for this feature:
- Treat eval.json as the source of truth for intended behavior.
- Derive integration test inputs and expectations from those cases where possible.
- If code and eval.json disagree, prefer eval.json and add a short comment in the test explaining the conflict.

Technical requirements:

1. Test structure and location
   - Integration tests MUST live under: /tests/integration at the project root.
   - All integration tests MUST be runnable via qtests-runner.mjs at the root (generated by the qtests module).
   - Use the qtests npm module for assertions and test harness behavior wherever possible; do NOT reimplement its functionality.

2. Server and app lifecycle
   - The server (or app instance) MUST be started at most ONCE for the entire integration test suite.
   - Use a shared setup file or beforeAll hook to:
     - start the server once,
     - reuse the same server/app instance across integration tests,
     - clean up after all tests complete.
   - Do not start/stop the server inside individual test cases.

3. Dependencies and mocking strategy
   - External network APIs and third-party HTTP services MUST be mocked or stubbed to avoid brittleness and external flakiness.
   - Use realistic mock responses consistent with eval.json expectations where applicable.
   - Core in-process modules and the application’s own data layer can be real (e.g., test database, in-memory DB, or local fixtures) as appropriate for integration testing.
   - The exception is qtests itself: always use the real qtests npm module as provided.

4. Assertions and expectations
   - Assert on stable contracts and invariants:
     - HTTP status codes,
     - required response fields,
     - key state changes in the database or application,
     - critical business rules (e.g., “user cannot access X without role Y”).
   - Avoid over-constraining unstable details like:
     - timestamps,
     - auto-generated IDs,
     - exact error message text, unless explicitly required by eval.json or the feature spec.
   - Prefer checking shapes and key fields over full deep equality of entire payloads.

5. Test readability and intent
   - Each test should clearly describe the workflow it exercises in its name and comments.
   - Group tests by workflow or feature (e.g., auth, billing, CRUD on important entities).
   - Where relevant, add a brief comment connecting the test back to the corresponding eval.json case or feature requirement.

Output:
- Generate the integration test files under /tests/integration with appropriate names and imports.
- Ensure all tests are discoverable and runnable by qtests-runner.mjs at the root without additional manual wiring.
